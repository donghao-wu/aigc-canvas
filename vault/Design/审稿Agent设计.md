# 审稿 Agent 设计文档

> 状态：**设计中**（未实现）
> 优先级：P0
> 见 [[Design/Agent扩展规划]]

## 需求背景

逐集生成后，剧本组需要人工审核每一集：
- 格式是否符合规范（△行/对白行/分行铁律）
- 人设是否漂移（角色行为与小传是否一致）
- 爽感是否达标（本集有没有值得追的内容）
- 节奏是否正确（开场钩/结尾悬念是否与大纲对应）

目前全靠人工，慢且容易遗漏。

## 输入 / 输出设计

### 输入

```json
{
  "mode": "review_episode",
  "episodeContent": "第N集完整剧本文本",
  "episodeIndex": N,
  "currentOutline": "第N集大纲行（含情节/钩子/结尾字段）",
  "characterBios": "角色小传（用于人设核对）",
  "storyBible": "故事圣经摘要（可选）"
}
```

### 输出（结构化）

```
【格式检查】⭐⭐⭐⭐☆
- △行比例：X%（规范：<14%）✓
- 发现问题：第3场第7行「林若曦拿起手机：我打电话」——动作混入对白行，应拆分

【人设核查】⭐⭐⭐☆☆
- 发现漂移：秦霍在本集第2场主动道歉，与小传「不轻易低头」性格矛盾
- 建议：改为被迫道歉或通过第三方传话

【爽感评估】⭐⭐⭐⭐⭐
- 本集爽点：林若曦当众揭穿陈可，台词「你以为我不知道吗」有力
- 节奏：开场钩（✓）→ 中段爽点（✓）→ 结尾悬念（✓）

【大纲匹配度】⭐⭐⭐⭐☆
- 情节吻合：✓
- 钩子对应：大纲写「电梯偶遇」，实际写成「停车场偶遇」，场景不同但效果相近
- 结尾悬念：✓

【总评】3.8/5 · 建议修改后入库
```

## System Prompt 草稿

```
你是专业的短剧剧本审稿编辑，从以下四个维度审核剧本：

【格式检查】
按照以下规范逐条核查，发现问题必须引用原文（"第X场第X行"）：
- 分行铁律：正文每行只能是△行或「人名：台词」，不能混写
- △行规范：不超过60字，不写环境描写
- 对白规范：同一角色不连续两行，情绪词只用规定词表
- 集末规范：最后一行必须是对白行

【人设核查】
对照角色小传，检查每个出场角色的行为、决策、台词是否符合其性格、动机和过往经历。
有漂移必须指出角色名 + 具体行为 + 违反的小传描述。

【爽感评估】
- 本集有没有爽点？爽点在哪一场？
- 开场钩是否在前3行内建立？
- 结尾悬念是否让人想看下一集？

【大纲匹配度】
对照提供的集数大纲，检查情节/钩子/结尾三个字段的吻合程度。
允许细节调整，但关键情节事件不得缺失或替换。

输出格式：四个维度各给1-5星评分 + 具体问题描述 + 修改建议，最后给总评分和建议。
禁止空泛表扬，必须具体指出问题或确认无问题。
```

## 前端实现方案

### 入口位置

在集数列表每集条目右键菜单 → "AI 审稿"

### UI 展示

在 `EpisodeContentView` 底部新增折叠面板"审稿报告"，流式输出审稿内容。

### API 接口

复用 `/api/script-agent`，新增 `mode: review_episode`：

```javascript
} else if (mode === 'review_episode') {
  const { episodeContent, episodeIndex, currentOutline, characterBios, storyBible } = req.body;
  systemContent = REVIEW_EPISODE_PROMPT;
  userContent = `请审核以下第${episodeIndex + 1}集剧本...`;
  // max_tokens: 2048
}
```

## 实现优先级排序

1. 格式检查（最容易实现，规则明确）
2. 大纲匹配度（次之，需要精确匹配字段）
3. 爽感评估（需要 model 有较强判断力）
4. 人设核查（最复杂，容易误判）
